# 소속 관련 시나리오 모음

> 이 문서는 소속(user_sdwt_prod)과 관련된 상황을 비개발자 관점에서 이해하기 쉽게 정리한 안내서입니다.

## 먼저 알아두기 (간단 용어)
- **소속(user_sdwt_prod)**: 사용자가 속한 그룹을 나타내는 값입니다.
- **승인자**: 소속 변경을 승인할 수 있는 사람(멤버/매니저)입니다.
- **관리자**: superuser/staff처럼 항상 승인/관리 권한이 있는 사람입니다.
- **예측 소속**: 외부 시스템이 추정한 소속 값입니다.
- **승인 대기**: 소속 변경 요청이 승인자를 기다리는 상태입니다.
- **재확인**: 예측 소속이 바뀌었을 때 사용자가 다시 선택하도록 안내하는 절차입니다.

## 시나리오 1: 신규 사용자 첫 로그인 (예측 소속 자동 적용)
상황: 처음 로그인한 사용자가 있고, 예측 소속이 존재합니다.
흐름:
1) 시스템이 외부 예측 소속을 확인합니다.
2) 예측 소속이 유효한 옵션이면 자동으로 소속을 적용합니다.
3) 본인 소속에 대한 기본 접근 권한이 자동으로 보장됩니다.
결과: 사용자는 별도 선택 없이 소속이 설정됩니다.

## 시나리오 2: 신규 사용자 첫 로그인 (예측 소속이 없거나 유효하지 않음)
상황: 예측 소속이 없거나, 선택 가능한 소속 목록에 없습니다.
흐름:
1) 시스템이 자동 적용을 하지 않습니다.
2) 사용자는 소속 선택(온보딩) 화면으로 안내됩니다.
결과: 사용자가 직접 소속을 선택해야 합니다.

## 시나리오 3: 소속이 없는 사용자의 온보딩
상황: 사용자에게 현재 소속이 없고, 승인 대기 요청도 없습니다.
흐름:
1) 시스템이 선택 가능한 소속 목록을 보여줍니다.
2) 사용자가 소속을 선택해 제출합니다.
3) 승인자 존재 여부와 예측값 일치 여부에 따라 자동 적용 또는 승인 대기로 나뉩니다.
결과: 소속이 즉시 적용되거나 승인 대기로 전환됩니다.

## 시나리오 4: 소속 변경 요청 (자동 적용)
상황: 사용자가 소속 변경을 요청합니다.
자동 적용 조건:
- 예측 소속과 동일한 값을 선택한 경우, 또는
- 해당 소속을 승인할 사람이 없는 경우
흐름:
1) 요청이 생성됩니다.
2) 시스템이 즉시 소속을 변경하고 적용 완료로 표시합니다.
결과: 승인 대기 없이 소속이 바로 바뀝니다.

## 시나리오 5: 소속 변경 요청 (승인 대기)
상황: 승인자가 존재하고, 예측 소속과 다른 값을 선택했습니다.
흐름:
1) 요청이 생성됩니다.
2) 상태는 승인 대기(PENDING)가 됩니다.
3) 승인자가 확인 후 승인/거절합니다.
결과: 승인 전까지 소속은 바뀌지 않습니다.

## 시나리오 6: 승인자가 요청을 승인
상황: 승인자가 대기 중인 요청을 승인합니다.
흐름:
1) 사용자의 소속/부서/라인 정보가 새 값으로 갱신됩니다.
2) 변경 요청 상태가 승인/적용 완료로 변경됩니다.
3) 이전 소속에 대한 권한이 자동으로 조정될 수 있습니다.
결과: 소속이 새 값으로 확정됩니다.

## 시나리오 7: 승인자가 요청을 거절
상황: 승인자가 요청을 거절합니다.
흐름:
1) 요청 상태가 거절로 변경됩니다.
2) 필요 시 거절 사유가 저장됩니다.
결과: 소속은 기존 값으로 유지됩니다.

## 시나리오 8: 대기 중 재요청(대체)
상황: 기존 승인 대기 요청이 있는 상태에서 다시 소속 변경을 요청합니다.
흐름:
1) 기존 대기 요청은 “대체됨”으로 처리됩니다.
2) 새 요청이 생성됩니다.
결과: 가장 최신 요청만 유효하게 남습니다.

## 시나리오 9: 외부 예측 변경 → 재확인 플래그 발생
상황: 외부 시스템의 예측 소속이 변경되었습니다.
흐름:
1) 시스템이 예측 변경을 감지합니다.
2) 현재 소속이 있고, 승인 대기 요청이 없으면 재확인 필요 상태로 표시합니다.
결과: 사용자는 재확인 안내를 받습니다.

## 시나리오 10: 재확인에서 기존 소속 유지
상황: 재확인 안내를 받은 사용자가 “기존 소속 유지”를 선택합니다.
흐름:
1) 재확인 플래그가 해제됩니다.
2) 소속은 그대로 유지됩니다.
결과: 추가 승인 없이 상태가 정상으로 돌아옵니다.

## 시나리오 11: 재확인에서 예측 소속 적용
상황: 재확인 안내에서 예측 소속을 그대로 선택합니다.
흐름:
1) 소속 변경 요청이 생성됩니다.
2) 예측값과 일치하므로 자동 적용이 가능합니다.
결과: 소속이 바로 변경될 수 있습니다.

## 시나리오 12: 재확인에서 다른 소속 선택
상황: 재확인 안내에서 예측값과 다른 소속을 선택합니다.
흐름:
1) 소속 변경 요청이 생성됩니다.
2) 이 경우는 승인 대기로 처리됩니다.
결과: 승인자 확인 후 소속이 변경됩니다.

## 시나리오 13: 접근 권한 부여
상황: 매니저가 다른 사용자에게 소속 접근 권한을 부여합니다.
흐름:
1) 대상 사용자에게 viewer/member/manager 역할 중 하나가 부여됩니다.
2) 본인 소속에 viewer를 부여하더라도 member로 자동 상향될 수 있습니다.
결과: 대상 사용자가 해당 소속 데이터를 볼 수 있게 됩니다.

## 시나리오 14: 접근 권한 회수 제한
상황: 매니저가 권한을 회수하려고 합니다.
제한 사항:
- 사용자의 “현재 소속” 권한은 제거할 수 없습니다.
- 마지막 manager를 제거하는 요청은 거부됩니다.
결과: 안전장치로 인해 일부 회수 요청은 실패할 수 있습니다.

## 시나리오 15: 멤버/요청 화면 보기
상황: 계정의 멤버/요청 화면을 엽니다.
흐름:
1) 현재 소속 메일함의 멤버 목록을 불러옵니다.
2) 승인 대기 요청 목록을 함께 표시합니다.
3) 승인 권한이 없는 사용자는 승인 버튼이 비활성화됩니다.
결과: 멤버와 요청을 한 화면에서 확인할 수 있습니다.

## 시나리오 16: 메일함 접근 요약
상황: 계정 개요 화면에서 메일함 접근 상태를 확인합니다.
흐름:
1) 일반 사용자는 본인이 접근 가능한 소속 메일함만 보입니다.
2) 관리자(superuser/staff)는 전체 메일함을 볼 수 있습니다.
3) UNASSIGNED(미분류) 메일함은 권한에 따라 숨겨질 수 있습니다.
결과: 권한 수준에 맞는 메일함 요약이 제공됩니다.

## 시나리오 17: assistant 접근 범위
상황: assistant 기능이 접근 가능한 소속 범위를 계산합니다.
흐름:
1) 시스템이 사용자의 접근 가능한 소속 집합을 계산합니다.
2) assistant 기능은 이 범위 내에서만 동작합니다.
결과: 소속 범위를 벗어난 데이터 접근이 제한됩니다.

## 시나리오 18: 라인별 소속 옵션 조회
상황: 라인 기준으로 선택 가능한 소속 목록을 보고 싶습니다.
흐름:
1) 시스템이 line별 user_sdwt_prod 목록을 제공합니다.
2) 사용자는 라인에 맞는 소속을 선택합니다.
결과: 라인 기준 선택이 가능해집니다.

## 시나리오 19: 계정 개요 화면 통합 보기
상황: 사용자가 “내 계정” 화면을 엽니다.
흐름:
1) 사용자 정보, 소속 정보, 변경 이력, 관리 가능한 그룹이 함께 로드됩니다.
2) 메일함 접근 요약과 재확인 상태도 함께 표시됩니다.
결과: 소속 관련 상태를 한 번에 파악할 수 있습니다.

## 참고: 자주 발생하는 예외 상황
- 승인 권한이 없는 사용자는 승인 대상 목록을 열람할 수 없거나 제한됩니다.
- 재확인 대상이 아닌 경우에는 재확인 응답이 거부됩니다.
- 승인 대기 요청이 있으면 자동 적용이 중단되고 승인 대기로 유지됩니다.
